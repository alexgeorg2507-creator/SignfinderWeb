## Severity-уровни

| Уровень | Что | Пример | Время реакции |
|---------|-----|--------|-----------------|
| **Sev1** | Прод недоступен ИЛИ подозрение на утечку чужих данных | 500 на всех эндпоинтах, юзер видит данные другого юзера | Немедленно, бросить всё |
| **Sev2** | Функция сломана, но прод работает | Не подписывается договор, но логин работает | В течение дня |
| **Sev3** | Косметика, не блокирует использование | Кривая вёрстка на 380px | Когда руки дойдут |

Не превращать Sev3 в Sev1 из тревожности. Не превращать Sev1 в Sev3 из лени.

## Где смотреть логи

```
# Cloud Run логи (API)
gcloud run services logs read signfinder-api --project=signfinder-prod --limit=100

# Только ошибки
gcloud run services logs read signfinder-api --project=signfinder-prod \
  --log-filter="severity>=ERROR" --limit=50

# Cloud SQL логи
gcloud sql operations list --instance=signfinder-db --project=signfinder-prod
```

Или через консоль: https://console.cloud.google.com/logs → фильтр по проекту.

**Правило:** логи НЕ должны содержать текст договоров и персональные данные
(ADR-007, ИБ-чеклист). Если видишь в логах текст документа — это баг,
заводить как Sev2 немедленно (утечка через логи — тоже утечка).

## Проверка на IDOR-атаку (кто-то пытается читать чужие данные)

Признак: много 404 на `/v1/me/*` с одного IP, или один юзер делает запросы
с разными паттернами id.

```
gcloud run services logs read signfinder-api --project=signfinder-prod \
  --log-filter='httpRequest.status=404 AND textPayload:"me/"' --limit=100
```

Если видишь systematic перебор — это не баг клиента, это разведка.
IDOR технически невозможен (tenant_id только из JWT, ADR-007), но если
кто-то долбит с попытками подмены — стоит посмотреть паттерн IP и, если
нужно, забанить на уровне Cloud Armor (не настроено на MVP-стадии, добавить
при первом реальном инциденте, не превентивно).

## Проверка на prompt injection (аномальное поведение LLM-пайплайна)

Признак: странные якоря в ответах `/v1/analyze`, документ с подозрительно
длинным временем обработки, JSON-схема validation failures растут.

```
gcloud run services logs read signfinder-api --project=signfinder-prod \
  --log-filter='textPayload:"schema validation failed"' --limit=50
```

Защита уже встроена (ADR-007): якоря сверяются с фактическим текстом PDF,
невалидный JSON отбрасывается. Рост числа отбрасываний — не инцидент сам
по себе, а сигнал что кто-то целенаправленно тестирует пайплайн. Смотреть
не паниковать, если сверка отрабатывает штатно.

## Rate limit / abuse

```
gcloud run services logs read signfinder-api --project=signfinder-prod \
  --log-filter='httpRequest.status=429' --limit=100
```

Много 429 с одного IP/юзера — либо баг клиента (retry-петля), либо
целенаправленный abuse лимитов. Смотреть паттерн, при необходимости —
временный бан на уровне Cloud Run (allow-list/deny-list) или снижение лимита.

## Что делать при Sev1 (прод недоступен)

1. `gcloud run services describe signfinder-api --project=signfinder-prod`
   — проверить статус, последний revision
2. Логи ошибок за последние 30 минут (см. выше)
3. Если после недавнего деплоя — rollback немедленно (GIT_WORKFLOW.md,
   раздел Rollback), разбираться в причине уже после отката
4. Если Cloud SQL недоступен — проверить `gcloud sql instances describe`,
   возможно квота/биллинг (см. ENVIRONMENTS_AND_COST.md — не кончился ли credit)
5. Записать в историю инцидентов ниже
