# SignfinderLand v2.0.0 — Deal Cycle

> Спека версии, замыкающей бизнес-процесс: инициатор подписал → передал ссылку
> контрагенту через мессенджер → контрагент подписал по ссылке без регистрации.
> Блокер go-live. Без этой версии продукт продаёт только половину ценности.
>
> Версионирование:
> - `SignfinderLand/version.txt` = 1.0 → 2.0.0
> - `signfinder-core` не меняется (актуальная — 1.20.18)
> - `signfinder-api` не меняется (актуальная — 1.18.4)
>
> Deal Cycle реализуется целиком в SignfinderLand backend + фронт, поверх
> существующих `/v1/analyze` и `/v1/sign` из signfinder-api. Ядро stateless.
> Major bump обоснован сменой позиционирования продукта, а не размером
> кодовой базы.

---

## 0. Ключевое решение — без backend-рассылок

**SignFinder не отправляет писем и сообщений. Точка.**
Инициатор получает временную ссылку → передаёт её контрагенту сам через любой
канал (WhatsApp/Telegram/email/SMS/бумажку с QR-кодом). Backend только
генерирует ссылку и следит за статусом сделки. Статусы инициатор видит в
своём кабинете при заходе.

Почему так:
- Никто не читает email от неизвестного сервиса — уходит в спам или игнор
- Письмо от знакомого («Вася, вот договор аренды: {ссылка}») открывают в 10 раз
  чаще чем от `no-reply@signfinder.app`
- Ссылка в чате мессенджера — часть истории разговора, к ней можно вернуться
- Нулевая инфраструктура почты: ни Postmark, ни SES, ни SPF/DKIM, ни $15/мес
- Юридически проще: SignFinder не оператор рассылок, только генератор ссылки —
  Privacy Policy упрощается радикально
- Виральность встроена: ссылка живёт в мессенджере как «место где мы подписали»,
  контрагент видит инструмент в работе и сам возвращается за собственной сделкой

Побочный эффект: пока нет мобильной версии кабинета — инициатору нужно
физически открывать веб-кабинет, чтобы видеть статус. Это осознанный
компромисс MVP; альтернатива (собственный email-канал уведомлений) хуже
описанных выше причин.

## 1. Позиционирование

**Было (SignfinderLand v1.x):** «инструмент подписи» — конкурент DocuSign в
узкой нише. Ценность заканчивается на моменте «инициатор скачал подписанный
PDF».

**Стало (SignfinderLand v2.0.0):** «инструмент закрытия сделки» —
противоположный конец рынка от DocuSign. Целевой сценарий: две стороны уже
договорились устно (аренда, услуги, микро-B2B), хотят зафиксировать без
ФНС, без нотариуса, без DocuSign-подобного тяжеловеса. Три клика, три
минуты, оба с PDF.

**Юр. уровень:** Простая электронная подпись (ПЭП) без регистрации второй
стороны. Достаточно для сделок между сторонами, которые друг другу доверяют
(это подтверждено 10 знакомыми владельца, использующими продукт).

## 2. Скоуп

### В скоупе

1. Модель сделки (Deal) со статусами и сроком хранения
2. Публичная страница подписания без регистрации
3. Ввод подписи второй стороной: фото с камеры / загрузка файла / рисование на canvas
4. Передача ссылки контрагенту: 3 кнопки в кабинете
   (Скопировать / Открыть в Telegram / Открыть в WhatsApp)
5. Раздел «Мои сделки» в кабинете со статусами и аудит-логом
6. Ретенция: 7 дней в кабинете, потом автоудаление PDF
7. Юридический след: ПЭП-дисклеймеры, IP/UA-логи, блок в финальном PDF
8. Опросник «Что улучшить?» → отправка в личный Telegram владельца
9. Скачивание финального PDF: и инициатор (из кабинета), и контрагент
   (со своей публичной страницы) — только через кнопку «Скачать»

### Явно вне скоупа

| Не делаем | Почему |
|-----------|--------|
| **Любые backend-отправки email** — уведомления инициатору, копии контрагенту, «истекает через 24ч» | См. §0 — принципиальное решение MVP |
| **SMTP / Postmark / SES интеграция** | Следствие §0 |
| **Push / mobile notifications** | Нет мобильного приложения |
| Multi-signer (>2 сторон) | Отложено, редкий сценарий для целевой аудитории |
| Напоминания контрагенту если не подписал | Инициатор сам напомнит в мессенджере |
| Отмена сделки после отправки | Ждём фидбек первых клиентов |
| SMS-подтверждение (усиление ПЭП) | При первом запросе клиента со сценарием «в суде» |
| Массовая отправка нескольким контрагентам | Не в целевом сценарии |
| Multi-tenant, роли, RBAC | Всё ещё single-tenant per user |
| Изменения в signfinder-core / signfinder-api | Не требуются, используем имеющиеся эндпоинты |
| mailto: как отдельная кнопка отправки | Инициатор при желании скопирует ссылку в почту сам — не хотим намекать на email как канал первого выбора |

## 3. Пользовательский сценарий (end-to-end)

**Инициатор (владелец аккаунта):**

1. Загружает договор → анализ → подписывает свою сторону *(существующий флоу v1.x)*
2. **Новое:** после подписания видит блок «Передать контрагенту»
3. Видит 3 кнопки:
   - **📋 Скопировать ссылку** — ссылка ложится в буфер, можно вставить куда угодно
   - **✈️ Открыть в Telegram** — открывается нативный `t.me/share/url?...` с готовым текстом
   - **📱 Открыть в WhatsApp** — открывается `wa.me/?text=...` с готовым текстом
4. Возвращается в «Мои сделки» — статус `sent` (переходит по клику любой кнопки)
5. Заходит в кабинет когда угодно проверить статус — `viewed` / `signed` / `expired`
6. При `signed` — в drill-down сделки кнопка «Скачать финальный PDF»

**Контрагент (без аккаунта):**

1. Получает ссылку в мессенджере от знакомого инициатора
2. Открывает — публичная страница с PDF-документом (с подписью инициатора)
3. Читает документ, видит дисклеймер ПЭП
4. Ставит обязательный чекбокс «Согласен с использованием ПЭП»
5. Выбирает способ подписи:
   - **Фото с камеры** — по умолчанию на мобилке
   - **Загрузить файл** — если подпись уже отсканирована
   - **Нарисовать** — canvas на touch-устройстве или мышью
6. Нажимает «Подписать»
7. Видит финальный PDF, кнопку «Скачать финальный PDF»
8. Сохраняет себе на устройство. Пока сделка активна (до 7 дней) — может вернуться
   по той же ссылке и скачать снова

## 4. Модель данных

Таблица `deals` в Postgres (SignfinderLand):

```sql
CREATE TABLE deals (
    id                          UUID PRIMARY KEY,
    initiator_user_id           UUID NOT NULL REFERENCES users(id),
    created_at                  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    expires_at                  TIMESTAMPTZ NOT NULL,  -- created_at + 7 дней
    status                      TEXT NOT NULL,         -- см. энум ниже
    share_token                 VARCHAR(32) UNIQUE NOT NULL,  -- nanoid, публичная ссылка
    share_channel_used          TEXT,                  -- copy_link | telegram | whatsapp | null (не жал)
    original_pdf_path           TEXT NOT NULL,         -- storage путь
    initiator_signed_pdf_path   TEXT NOT NULL,
    final_pdf_path              TEXT,                  -- null пока не подписан контрагентом
    saved_anchors               JSONB NOT NULL,        -- якоря от первого analyze (для контрагента)
    audit_log                   JSONB NOT NULL DEFAULT '[]',
    counterparty_signature_meta JSONB
);

CREATE INDEX ix_deals_initiator ON deals(initiator_user_id, created_at DESC);
CREATE UNIQUE INDEX ix_deals_share_token ON deals(share_token);
CREATE INDEX ix_deals_expires ON deals(expires_at) WHERE status IN ('draft','sent','viewed');
```

**Внимание — ADR-002 (договоры не персистятся) частично уточняется:**
Deal Cycle требует хранить PDF в течение срока действия сделки (7 дней).
Это осознанное отступление от ADR-002 в рамках конкретного сценария —
storage путь ограничен `deals/{deal_id}/`, автоудаление по ретенции. Оформить
отдельным ADR как «уточнение к ADR-002 для сценария Deal Cycle».

**Что убрано из схемы** (по сравнению с более ранним драфтом): `counterparty_channel`
и `counterparty_contact` — не собираем email/телефон контрагента вообще,
потому что не связываемся с ним от лица SignFinder ни по какому каналу.
`counterparty_email` — тоже убрано, копий контрагенту не шлём.
`share_channel_used` оставлен для аналитики: понимать какой канал
популярнее у пользователей, не более.

**Статусы (enum на уровне Python, TEXT в БД):**

| Статус | Когда |
|--------|-------|
| `draft` | Инициатор подписал свою часть, ещё не нажал ни одну кнопку передачи |
| `sent` | Инициатор нажал «Скопировать / Открыть в TG / Открыть в WA» — считаем что ссылку передал |
| `viewed` | Контрагент открыл ссылку хотя бы раз |
| `signed` | Контрагент подписал, финальный PDF сгенерирован |
| `expired` | `expires_at` наступил, статус ≠ `signed` |
| `rejected` | Резерв, пока не используется |

**Формат `audit_log`:**

```json
[
  {"event": "created",  "at": "2026-07-24T10:15:00Z", "actor": "initiator"},
  {"event": "sent",     "at": "2026-07-24T10:16:12Z", "channel": "whatsapp"},
  {"event": "viewed",   "at": "2026-07-24T14:22:03Z", "ip": "1.2.3.4", "ua": "Mozilla/..."},
  {"event": "signed",   "at": "2026-07-24T14:24:41Z", "ip": "1.2.3.4", "ua": "Mozilla/..."}
]
```

**Storage layout:**

```
deals/{deal_id}/original.pdf
deals/{deal_id}/initiator_signed.pdf
deals/{deal_id}/final.pdf            ← после подписи контрагента
```

Хранение PDF — в существующем StorageBackend (LocalFS/GCS), не в БД.

## 5. API эндпоинты (в SignfinderLand backend)

### Приватные (за Firebase Auth)

| Метод | Путь | Что |
|-------|------|-----|
| POST | `/v1/deals` | Создать сделку из подписанного инициатором PDF |
| GET | `/v1/deals` | Список сделок инициатора (пагинация) |
| GET | `/v1/deals/{id}` | Детали + аудит-лог |
| POST | `/v1/deals/{id}/mark-shared` | Отметить что инициатор нажал кнопку передачи (`sent` + channel_used) |
| GET | `/v1/deals/{id}/final-pdf` | Скачать финальный PDF (для инициатора) |
| POST | `/v1/feedback` | Опросник → Telegram владельца |

*Убран `POST /v1/deals/{id}/send` — раз backend не шлёт письма, нечего дёргать.*

### Публичные (без auth, по `share_token`)

| Метод | Путь | Что |
|-------|------|-----|
| GET | `/v1/public/deals/{share_token}` | Метаданные для страницы подписания |
| GET | `/v1/public/deals/{share_token}/pdf` | Скачать PDF со стороной инициатора |
| POST | `/v1/public/deals/{share_token}/sign` | Подписать (payload: подпись + чекбокс ПЭП) |
| GET | `/v1/public/deals/{share_token}/final-pdf` | Скачать финальный PDF (для контрагента, после `signed`) |

**Rate limit публичных:** 10 req/min на share_token, 60 req/min на IP.

## 6. Позиционирование подписи контрагента

**Решение:** переиспользуем якоря первого анализа, не гоняем LLM повторно.

При создании Deal (шаг 1) в `saved_anchors` пишутся **все** якоря из первого
`/v1/analyze`, включая ту сторону, которую инициатор уже подписал. Для
контрагента фильтруются якоря противоположной стороны по `party_role`.

Плюсы:
- 0 токенов LLM на этап контрагента
- Идентичное позиционирование с ожиданием инициатора («я вижу — куда он должен подписать»)
- Работает быстро (нет ожидания LLM на публичной странице)

**Ограничение:** если у документа только одна сторона в якорях (плохой
анализ) — контрагенту не найдётся места. Fallback: показать контрагенту
PDF и предложить «нарисовать где расписаться» — реализация в v2.1, не сейчас.

## 7. Юридический след

**В финальном PDF** — блок на новой странице в конце (или в футере последней
страницы если помещается):

```
─────────────────────────────────────────────────────────
Электронная подпись сформирована через SignFinder (signfinder.app).
Простая электронная подпись (ПЭП) по соглашению сторон.

Инициатор: {email}
  Подписано: {timestamp UTC}, IP {ip}, UA {ua_shortened}

Контрагент подписал по временной ссылке:
  Подписано: {timestamp UTC}, IP {ip}, UA {ua_shortened}
  Способ ввода подписи: {camera|file|canvas}

ID сделки: {deal_id}
─────────────────────────────────────────────────────────
```

**Соглашение об ПЭП** — отдельная страница на лендинге + ссылка в футере
кабинета + чекбокс на публичной странице.

**Обязательный чекбокс контрагента** на публичной странице:
> «Я согласен что моя подпись, поставленная в этом документе, является
> простой электронной подписью в понимании ст. 5 ФЗ-63, и признаю её
> равнозначной собственноручной подписи на бумажном документе.»

(Формулировка юридически окончательная — по фидбеку юриста; сейчас черновик.)

## 8. Этапы разработки

### E1 — Модель Deal + миграция (1-2 дня)

- Alembic-миграция `deals`
- Pydantic-модели: `Deal`, `DealCreate`, `DealPublicView`, `DealSignRequest`
- Расширить storage layout
- Приватные API: `POST /v1/deals`, `GET /v1/deals`, `GET /v1/deals/{id}`
- Unit-тесты моделей и API (с моком storage)

### E2 — Публичная страница подписания (3-4 дня)

- Публичный роут фронта: `/sign/{share_token}` — без auth
- Публичные API: `GET /public/deals/{token}`, `GET .../pdf`, `POST .../sign`,
  `GET .../final-pdf`
- PDF viewer (переиспользуем существующий из кабинета)
- Дисклеймер ПЭП + обязательный чекбокс
- Три способа ввода подписи:
  - фото: `<input type="file" accept="image/*" capture="environment">`
  - файл: обычный upload
  - canvas: HTML5 canvas с pointerdown/pointermove
- Обработка подписи через существующий `POST /v1/signers/{tmp_id}/signature/process`
  из `signfinder-api` (создаём временного signer, не сохраняем)
- Наложение подписи на `initiator_signed.pdf` через `POST /v1/sign` с якорями
  противоположной стороны
- Генерация финального PDF с юридическим блоком
- Кнопка «Скачать финальный PDF» после подписания
- Rate limits

### E3 — Передача ссылки из кабинета (1 день)

Резко упростилось после отказа от backend-рассылок.

- В UI после подписания инициатором — блок «Передать контрагенту»
- 3 кнопки:
  - **📋 Скопировать ссылку** — `navigator.clipboard.writeText(shareUrl)`
  - **✈️ Открыть в Telegram** — `https://t.me/share/url?url={link}&text={text}`
  - **📱 Открыть в WhatsApp** — `https://wa.me/?text=${text+link}`
- Клик любой кнопки → `POST /v1/deals/{id}/mark-shared` с `channel_used`
- Backend в мессенджерах не участвует. Никаких email-провайдеров, SMTP, DNS —
  нет вообще.
- Текст сообщения: «{имя инициатора если указано, иначе email} направил вам
  договор на подпись через SignFinder. Откройте ссылку, ознакомьтесь и
  подпишите: {link}»

### E4 — Кабинет сделок (1-2 дня)

Упростилось после отказа от email-уведомлений.

- Раздел «Мои сделки» в UI
- Таблица: дата, документ, статус, срок, канал передачи
- Drill-down: аудит-лог, ссылка на финальный PDF (если `signed`),
  кнопка «Скопировать ссылку ещё раз» (если сделка активна и не подписана)
- Опционально (nice-to-have, не блокер): бейдж в topbar «(N)» с числом
  сделок с новыми событиями с прошлого визита — если не успеваем в E4,
  переносим в v2.1

### E5 — Ретенция и cleanup (1 день)

- APScheduler в SignfinderLand backend, раз в час:
  - помечает `expired` где `expires_at < now()` и status ∈ (`draft`,`sent`,`viewed`)
  - удаляет PDF-файлы из storage у `expired` записей
  - оставляет запись Deal (без файлов) для аудита ещё 30 дней
- Cron ежедневно в 03:00 UTC:
  - удаляет Deal полностью где `expires_at + 30d < now()`

### E6 — Опросник → Telegram (0.5 дня)

- Пункт меню в кабинете: «Что улучшить?»
- Форма: одно поле «За какую фичу вы готовы заплатить?» + опциональный контакт
- API `POST /v1/feedback` → Telegram Bot API `sendMessage`
- Env: `TG_FEEDBACK_BOT_TOKEN`, `TG_FEEDBACK_CHAT_ID`
- В UI: «Спасибо, передано разработчику»

### E7 — Юр. след + мобильная косметика (1-2 дня)

- Соглашение об ПЭП: отдельная страница на лендинге + ссылки в футере + чекбокс
- Юр. блок в финальном PDF (см. §7)
- Мобильный UX публичной страницы: тач-цели ≥44px, камера работает на iOS Safari
- Ручные тесты на iPhone Safari + Android Chrome
- Обновить `SignfinderLand/version.txt` → `2.0.0`

**Итого: 8-13 рабочих дней.** (Сократилось с 10-15 после решения об
отказе от backend-рассылок.)

## 9. Критерии готовности версии

1. Владелец подписывает свой договор аренды и передаёт арендатору **за ≤3 клика**
   (подписать → скопировать → вставить в WhatsApp)
2. Арендатор с телефона без регистрации подписывает **за ≤5 кликов и ≤2 минут**
3. Оба могут скачать финальный PDF (инициатор — из кабинета, контрагент —
   со своей публичной страницы)
4. Владелец видит актуальный статус сделки в кабинете при заходе
5. Работает на iPhone Safari и Android Chrome (ручной тест)
6. Опросник доставляет сообщение в личный TG владельца
7. Истёкшие сделки корректно чистятся через 7 дней (PDF), через 30 дней (запись)
8. Финальный PDF содержит юр. блок с IP/UA/timestamps обеих сторон
9. Публичная страница не отдаёт данных инициатора кроме email
   (для отображения «Договор от {email}»)
10. Бейдж версии в topbar кабинета показывает `v2.0.0+{BUILD_NUMBER}`
    (см. `TASK_versioning.md`)
11. **Ни одной backend-рассылки email никому и никогда** — проверить в коде
