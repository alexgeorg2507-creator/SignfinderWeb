## ADR-001 — Разделение Cloud / Enterprise как два продукта, не один

**Дата:** июнь 2026
**Статус:** Принято

**Контекст:** Один лендинг пытался говорить с обоими сегментами (ИП + агентства).
Слабее, чем два заточенных — сегменты живут по разным моделям доверия к данным.

**Решение:** Два лендинга, два позиционирования. Cloud — self-service, sandbox,
тёплый дизайн. Enterprise — demo-call, локальная установка, тёмный дизайн.

**Последствия:** Два набора текстов/дизайна на поддержке. Оправдано — конверсия
важнее экономии на копирайтинге.

## ADR-002 — Договоры пользователей НЕ персистятся

**Дата:** июнь 2026
**Статус:** Принято (уточнено ADR-009 для v2.0.0 Deal Cycle)

**Контекст:** Кабинет обрабатывает чужие договоры (PDF/DOCX). Хранить — значит
нести GDPR-ответственность за чужие юридические документы.

**Решение:** Обработка только in-memory. Cloud Run получил байты → обработал →
вернул PDF → забыл. Ничего не пишется на диск, в БД, в GCS.

**Последствия:** Нет истории документов в кабинете (фича, не баг — можно добавить
позже, если попросят). Радикально снижена поверхность утечки.

**Что храним вместо этого:** только подпись пользователя (bytea) и один шаблон
стороны — минимальный набор для работы продукта.

## ADR-003 — Аутентификация: Firebase Auth + свой Postgres (гибрид)

**Дата:** июнь 2026
**Статус:** Принято

**Контекст:** Нужен вход с verified email. Варианты: полностью свой auth (пароли,
токены, письма) или полностью Firebase (auth + Firestore).

**Решение:** Firebase Auth только для идентификации (Google + email + verification).
Данные пользователя (профиль, подпись, шаблон, usage) — в собственном Postgres.

**Обоснование:** Не изобретаем аутентификацию — это решённая задача, Google делает
её лучше. Но контролируем схему данных сами — не привязаны к Firestore
NoSQL-модели, можем делать нормальные foreign keys и транзакции (см. ADR-005).

**Последствия:** Два разных Firebase-проекта под test/prod (см. ADR-004) означают
раздельный OAuth consent screen setup — на каждую среду отдельная настройка.

## ADR-004 — Две изолированные GCP-среды (test / prod)

**Дата:** июнь 2026
**Статус:** Принято

**Контекст:** Разработка через AI-агента на прямом доступе к прод-инфраструктуре —
неприемлемый риск после появления реального трафика.

**Решение:** `signfinder-test` и `signfinder-prod` — два независимых GCP-проекта.
Разные Cloud SQL, разные Firebase Auth, разные секреты. Общий образ Docker,
промоутится test → prod вручную.

**Обоснование:** Не Terraform, не полноценный IaC — это оверинжиниринг для
одного разработчика и AI-агента. Но физическая изоляция сред — не оверинжиниринг,
это минимальная гигиена перед тем как пускать трафик на прод.

**Последствия:** Двойная стоимость Cloud SQL (см. ENVIRONMENTS_AND_COST.md).
Принято сознательно — цена ошибки на проде выше $10-15/мес разницы.

## ADR-005 — Подпись пользователя хранится как bytea в Postgres, не в GCS

**Дата:** июнь 2026
**Статус:** Принято

**Контекст:** Одна подпись на пользователя, ~50-200KB PNG. Развилка: bytea в уже
оплаченном Cloud SQL или отдельный GCS-бакет с ACL.

**Решение:** bytea в Postgres.

**Обоснование:** Cloud SQL инстанс оплачивается в любом случае (даже пустой).
Подписи в него — предельные издержки почти нулевые. GCS — новая статья расходов
плюс второй путь изоляции данных (ACL/signed URLs) для объёма, который не
оправдывает сложность.

**Путь миграции:** Если появится история документов или объём вырастет на порядки —
переезд в GCS на этапе REVIEW. Не блокирует текущую архитектуру.

## ADR-006 — tenant_id закладывается сейчас, даже когда равен user_id

**Дата:** июнь 2026
**Статус:** Принято

**Контекст:** Сейчас один пользователь = один тенант (нет команд/организаций).
Вопрос: делать поле `user_id` напрямую, или сразу вводить абстракцию `tenant_id`.

**Решение:** Все таблицы данных пользователя ссылаются на `tenant_id`, не
напрямую на `user_id`. Пока `tenant_id == user_id` для персональных аккаунтов.

**Обоснование:** Когда появятся команды/организации (несколько юзеров = один
тенант с ролями) — не нужна миграция схемы, только добавление записей в
таблицу `tenants` и `tenant_members`. Стоимость сейчас — одно дополнительное
поле. Стоимость отказа — болезненный рефакторинг на живых данных позже.

**Границы:** Это НЕ полноценная мультитенантность (нет orgs/RBAC/ролей) —
это задел под неё. Полноценная реализация — только когда появится реальный
спрос (компания с несколькими сотрудниками просит доступ).

## ADR-007 — Security model: три принципа против IDOR / CRUD-атак / prompt injection

**Дата:** июнь 2026
**Статус:** Принято

**Контекст:** Минимальный ИБ для мелкого B2B: защита от подмены ID, от
CRUD-злоупотреблений, от инъекций через содержимое документов в LLM-пайплайн.

**Решение:**
1. **Authorization derived from verified identity** — tenant_id только из
   подписанного Firebase JWT, никогда из тела запроса.
2. **Strict input contracts** — Pydantic `extra='forbid'`, JWT в заголовке
   (не в куках, CSRF неактуален by design), rate limiting.
3. **Untrusted document content** — текст договора идёт в LLM как недоверенный
   user-контент с явной обёрткой; вывод модели валидируется по JSON-схеме;
   якоря сверяются с фактическим текстом PDF (model output never trusted blind).

**Последствия:** Каждый новый эндпоинт обязан проходить эти три проверки —
формализовано как чеклист в TASK_MVP_Lite_Client.md, теперь дублируется
здесь как постоянное правило, не разовая задача.

## ADR-008 — CI/CD: агент не деплоит на prod, только человек кнопкой

**Дата:** июнь 2026
**Статус:** Принято

**Контекст:** До этого момента AI-агент напрямую гонял `gcloud deploy` на
единственную среду. С появлением прод-трафика и реальных пользовательских
данных это неприемлемый риск (см. историю "агент удалил прод базу").

**Решение:** Агент имеет право коммитить, открывать PR, гонять тесты и
деплоить на **test**. Деплой на **prod** — только вручную человеком после
зелёных тестов на test. См. GIT_WORKFLOW.md.

**Обоснование:** Не "агент плохой". Агент делает то, что ему разрешили.
Разрешать автономный прод-деплой без review — ошибка процесса, не агента.

## ADR-009 — Deal Cycle: PDF хранятся 7 дней в рамках сценария сделки (уточнение ADR-002)

**Дата:** июль 2026
**Статус:** Принято
**Связь:** уточняет ADR-002 в рамках v2.0.0 Deal Cycle, не отменяет его

**Контекст:** ADR-002 говорит «договоры НЕ персистятся». Deal Cycle требует
временного хранения: инициатор подписал → контрагент подписывает позже
(может через часы или дни), всё это время его PDF должен где-то лежать —
в памяти Cloud Run его не оставить между запросами.

**Решение:** PDF в рамках сделки хранятся в существующем StorageBackend
(LocalFS/GCS) по путям `deals/{deal_id}/{original|initiator_signed|final}.pdf`
в течение **7 дней** (`expires_at = created_at + 7d`). По истечении PDF
автоматически удаляются cron'ом, запись Deal без файлов остаётся ещё 30 дней
для аудита, затем удаляется целиком.

**Обоснование:** Асинхронность сценария — прямое следствие бизнес-ценности
«закрыть сделку, а не просто подписать» (см. DEAL_CYCLE_SPEC.md §1). Отказ от
временного хранения вообще — отказ от всей версии. 7 дней — сколько реально
лежат неподписанные типовые бытовые договоры (оценка по целевому сценарию
аренды / услуг фрилансеров).

**Последствия:**
- Новая точка GDPR-ответственности (временно храним чужие договоры) —
  Privacy Policy обязана прямо указывать 7-дневный цикл
- Стоимость GCS storage малая на MVP-объёмах (десятки-сотни PDF в месяц, ~1MB
  каждый, максимум ~200MB с учётом трёх версий)
- APScheduler-cron — новая зависимость SignfinderLand, надо в мониторинг
  (алерт если cron не отрабатывал >2ч)
- Начальный принцип ADR-002 (всё в памяти) остаётся для «одноразовых»
  режимов (не-Deal Cycle подписание в кабинете, sandbox на лендинге) — там
  по-прежнему ничего не персистится

## ADR-010 — Отказ от backend-рассылок email в v2.0.0

**Дата:** июль 2026
**Статус:** Принято

**Контекст:** При проектировании Deal Cycle рассматривался вариант: SignFinder
отправляет письма контрагенту (ссылка на подпись) и инициатору
(уведомления: viewed / signed / истекает через 24ч) через email-провайдера
(Postmark/SES/SendGrid). Предполагаемая стоимость — $15/мес после free tier,
DNS-настройка (SPF/DKIM) 1-2 дня, поддержка спам-репутации домена.

**Решение:** SignFinder не отправляет email никому в v2.0.0. Инициатор получает
временную ссылку и передаёт её контрагенту сам через любой канал (3 кнопки
в кабинете: Скопировать / Telegram / WhatsApp). Статусы инициатор видит в
кабинете при заходе (polling глазами). Копии финального PDF — только кнопка
«Скачать» (инициатор — из кабинета, контрагент — со своей публичной
страницы).

Исключение: Telegram Bot API для доставки опросника в личный чат владельца —
это не email-рассылка и не письмо от имени SignFinder к большому пулу
получателей, а один внутренний адресат.

**Обоснование:**
1. Письмо от неизвестного сервиса (`no-reply@signfinder.app`) контрагент чаще
   игнорирует или отправляет в спам, чем письмо/сообщение от знакомого
   в мессенджере.
2. Нулевая инфраструктура: нет email-провайдера, DNS-записей, секретов в
   Secret Manager, кода отправки в backend'е.
3. Виральность встроена сильнее: ссылка в чате мессенджера — часть истории
   разговора сторон, видна обоим как «место где мы подписали».
4. Юридически проще: SignFinder не оператор рассылок, меньше требований к
   Privacy Policy в части согласия на коммуникацию.

**Последствия:**
- Инициатор должен физически открывать кабинет чтобы видеть статус сделки.
  Компенсируется опциональным бейджем в topbar «(N) новых событий»
  (на усмотрение E4, можно перенести в v2.1)
- Контрагент ответственен сам скачать финальный PDF пока сделка
  активна (7 дней). Не скачал — нет второго шанса.
- Когда появятся мобильные пуш-уведомления (PWA или native) — это пересмотр.
  До того момента email как канал уведомлений не даёт достаточной ценности
  чтобы оправдать инфраструктуру.
- Принципиальный отказ — в коде не должно быть импортов SMTP-библиарий и
  email-шаблонов в принципе. Критерий готовности v2.0.0 (№ 11):
  «Ни одной backend-рассылки email никому и никогда — проверить в коде».
- См. DEAL_CYCLE_SPEC.md §0 — ключевое архитектурное решение версии.
