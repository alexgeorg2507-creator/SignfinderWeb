## Где смотреть

PostHog EU (проект один на обе среды — лендинг + кабинет пишут в один датасет,
фильтровать по `$current_url` или свойству окружения, если размечено).

## События — лендинг

| Событие | Когда | Что показывает |
|---------|-------|-----------------|
| `$pageview` | Открытие любой страницы | Трафик, источники |
| `cta_click` | Клик по любой CTA-кнопке (`button=` в свойствах) | Какие блоки лендинга реально работают |
| `lang_switch` | Переключение RU/EN | Актуальность EN (сейчас заморожен — если событие всё же есть, повод пересмотреть) |
| `sandbox_demo_click` | Клик "показать демо" в sandbox-секции | Вовлечённость в интерактив |
| `sandbox_demo_download` | Скачал демо-результат (мок-договор) | Досмотрел до конца воронки на лендинге |
| `cta_click button=dropzone_to_app` | Дропзона → переход в /app | Готовность попробовать с реальным документом |

### Воронка лендинга
```
$pageview → sandbox_demo_click → sandbox_demo_download → cta_click(→ /app or Telegram)
```
Enterprise-лендинг: тот же sandbox вторичен, основная CTA — "Запросить демо" → Telegram,
там воронка обрывается на переходе, дальше конверсия не в PostHog, а в Telegram-диалоге
(считать вручную).

## События — кабинет

| Событие | Когда |
|---------|-------|
| `signup_started` | Нажал «Войти» |
| `signup_completed` | Email verified, кабинет открыт |
| `profile_filled` | Заполнил профиль |
| `signature_uploaded` | Сохранил подпись |
| `document_signed` | Подписал договор |
| `limit_reached` | Достиг лимита (10 док/мес) |

### Воронка активации
```
signup_started → signup_completed → profile_filled → signature_uploaded → document_signed
```

Это главная метрика продукта. `document_signed` = активация. Всё, что раньше
этой точки — не считается успехом, сколько бы регистраций ни было.

## Что смотреть каждую неделю

1. **Воронка кабинета целиком** — где обрывается сильнее всего:
   - Обрыв на `signup_started → signup_completed` — проблема с Auth/email-verify UX
   - Обрыв на `signature_uploaded` — проблема с камерой/OpenCV-обработкой, самый
     хрупкий технический шаг
   - Обрыв после `signature_uploaded`, до `document_signed` — проблема с
     рабочим экраном (analyze/sign)
2. **Retention-сигнал**: сколько документов на активированного пользователя за
   период. 1 документ = попробовал разово. 3+ = зацепило.
3. **`limit_reached`**: если это событие вообще есть — сигнал спроса на платный
   тариф. Пока нет ни одного — pricing обсуждать рано (см. `BACKLOG.md`).
4. **Воронка лендинга**: соотношение `sandbox_demo_click` к `$pageview` —
   зацепляет ли sandbox вообще. Если низкое — проблема не в кабинете, а выше
   по воронке, в самом лендинге/трафике.

## Красные флаги — смотреть сразу, не ждать недели

- `signup_completed` есть, `profile_filled` почти нет за несколько дней подряд —
  что-то сломано на первом экране кабинета, не статистика, а баг
- Резкий скачок `limit_reached` без роста `signup_started` — либо баг счётчика
  (см. атомарность транзакции в `TECH_SPEC_landing_cabinet.md`), либо один
  агрессивный пользователь молотит API
- `document_signed` = 0 несколько дней при ненулевом трафике — тревога уровня
  Sev2 минимум, см. `RUNBOOK_TECH_SECURITY.md`

## Чего в PostHog НЕТ и не должно быть

Текст договоров, персональные данные пользователей (ФИО, реквизиты) —
события фиксируют только факт действия, не содержимое. Если увидел в событии
что-то похожее на текст документа или ФИО в свойствах — это баг трекинга,
заводить как утечку (ADR-007 / `RUNBOOK_TECH_SECURITY.md`), не игнорировать.
